Main Idea

GroceryHelper is a tool to make grocery shopping quicker and easier. When ingredients and an origin location are inputted, our GroceryHelper webapp will output a shopping plan which includes: a list of stores to visit in sequential order, a map with directions to each store and back to your origin, and a list of which ingredients to buy at each store. The outputted path is the minimum distance the user has to travel in order to get all the ingredients they need. The goal of GroceryHelper is to minimize time spent shopping and the disappointment of caused by not finding the items that the shopper wants at the grocery store. Additionally, if the shopper does not know what they want to eat they can input a cuisine and some ingredients and GroceryHelper will output a recipe and a path plan to purchase the extra ingredients needed.

Implementation information

The GroceryHelper path finding algorithm works by first filtering grocery stores by Euclidean distance away from the location of the user. This is done by using the latitude and longitude of the locations. It then identifies which stores have which ingredients and makes a list of every path possible to get all the ingredients. Each calculated path is then scored based on how many stops there are and what the total distance traveled is. The path with the minimum score is chosen and then the route the user should take along with the items they should buy at each store is given to the user.

Project evolution

Our initial goal for this project was to create a WebApp that could tell you the cheapest and fastest places to buy your groceries. Unfortunately, came up with the idea before we actually found an API, and, unfortunately, the majority of the information that we needed to be successful was very proprietary, and therefore could not be found for free on the interwebs. So instead of focusing on prices, we pivoted a little and decided to focus the app on finding the fastest route to get all of your groceries. An added feature was that we roped in a recipe database and added a meal planning helper. Thus, GroceryHelper was born.

Along the development path, we hit more road bumps. We started the project using Django for our database and planned to use it for web development as well. However, it turns out that Django is really confusing and complete overkill for the website applications we needed, so a few weeks in, we switched everything over to Flask. We had used Flask during other class assignments and felt comfortable with it, so things went much more smoothly from there. We love Flask!

Another problem arose when we started testing our path planning algorithm. After looking more closely at the data that we were getting back from the Supermarket API, we realized that many areas of the country were very poorly represented, and some not at all. We also found that the way that foods were classified was real weird. For example, if you searched a store for apples, it would first give you all of the different kinds of apple baby food that the store held. At that point in the project, we decided to just move forward with the API that we had, even though it had some data quality issues.

In the end, we ended up with this great product that can very reliably tell you the fastest ways to get to stores in your area to get baby food, but only if you live in a major metropolitan area. :)

Attribution

We used three APIs to create our app:

  1. Supermarket API for product information at local stores
  2. Yummly API for the recipe database
  3. Google Maps for geocoding, driving directions, and live map
